## 五、发现的问题

### ISSUE-R1: 系统内部超时截断（已修复，v1.1.0）

- **严重程度**: 已解决
- **现象**: 9/60 题 (15%) 输出 `⚠️ 分析超时`
- **根因**: `total_timeout=900` 硬编码在 CLI 中
- **修复**: v1.1.0 移除 total_timeout 限制，改为 CPU 卡死检测 + httpx 超时保护
- **验证**: 9 题全部重跑通过，RC=0，无截断

### ISSUE-R2: v4 普遍比 v3 慢

- **严重程度**: 低
- **现象**: 28/40 H 级题目比 v3 慢，平均 +292s
- **分析**: v4 修复后分析更完整（更多 SQL、更多验证步骤），属于质量换时间的 trade-off
- **v3 快但可能不够完整**：v3 的 H33 仅 388s/20 tools，v4 的 H33 978s 但仍触发内部超时
- **建议**: 持续监控，如果耗时持续增长需排查性能退化

### ISSUE-R3: 补查功能未触发（根因已定位并修复）

- **严重程度**: 低 → 已修复
- **现象**: 数据充分性检查（修复项 4）在 60 题测试中 0 次触发
- **根因**: `_extract_tables_from_sql` 方法使用 `sqlparse.flatten()` 后，`Identifier` 对象被拆成独立 Token，表名提取始终返回空集合，导致 Plan 表与 SQL 表无法正确匹配
- **修复**: 将 sqlparse 方案改为正则直接匹配 `schema.table` 模式，更简洁可靠
- **验证**: 6 项单元测试全部通过，端到端测试确认 plan-based 流程可达
- **实际触发条件**: Plan 中列出的表 ≠ SQL 中实际查询的表（两者由同一 LLM 生成，自然一致性高，仅在 LLM 遗忘时触发）

3. **降级+重试机制生效**：13 题降级、9 题重试，保障了整体成功率
4. **v4 分析更完整但更慢**：28/40 H 题比 v3 慢，平均多 292s，但修复了 v3 的 4 题超时和 3 题字体问题
